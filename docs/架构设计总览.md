# DAI 实验平台 — 架构设计与前后端对接总览

## 一、文档定位

本文档用于说明 DAI 实验平台的整体架构、前后端职责边界、接口规划、核心数据流与联调验收标准。它面向后端开发、前端开发和后续维护人员，重点服务前后端并行开发与接口对接。

当前仓库仍处于项目初始化阶段：

- `backend/` 已建立 FastAPI 后端目录骨架，但业务代码尚未实现。
- `frontend/` 当前仅保留占位文件，前端由另一位同学独立开发。
- 本文档中的 API、数据库实体、状态枚举均为后续实现规划，不代表当前接口已经可用。
- 后端实现后，FastAPI 自动生成的 Swagger 文档将作为前端对接的最终接口契约。

对接原则：

- 后端以 `/api/v1` 作为业务接口统一前缀。
- 前后端通过 JSON 请求与响应交互。
- 需要登录的接口统一使用 `Authorization: Bearer <access_token>`。
- 接口字段、枚举值、错误格式以 Swagger 文档和本文档保持一致为目标。

## 二、平台定位

DAI 实验平台是一个面向 **AI、机器学习、深度学习教学** 的在线实验平台。学生在浏览器中即可完成课程学习、代码实验、作业提交、自动判题、考试与成绩查看，无需在本地安装 Python、Jupyter 或深度学习环境。

核心目标是打通完整教学闭环：

```
理论学习 -> 在线实验 -> 作业提交 -> 自动判题 -> 成绩反馈 -> 教师分析
```

平台优先覆盖以下场景：

- 学生登录后查看课程、章节、课时与作业。
- 学生在网页代码编辑器中提交 Python 函数型编程题。
- 系统通过 Docker 沙箱执行隐藏测试并返回判题结果。
- 学生通过前端 iframe 进入 JupyterLab 完成交互式实验。
- 教师管理课程、题目、作业、考试与学生成绩。
- 管理员创建账号、分配角色、维护基础数据与系统参数。

## 三、目标用户与权限角色

| 角色 | 核心场景 | 默认权限 |
|------|----------|----------|
| 学生 `student` | 学习课程、查看课时、提交作业、参加考试、进入 JupyterLab、查看个人成绩 | 只读课程内容，提交自己的作业/考试 |
| 教师/助教 `teacher` | 创建课程、维护章节课时、布置作业与考试、查看学生提交和成绩 | 管理自己负责的教学内容 |
| 管理员 `admin` | 创建用户、分配角色、管理课程归属、配置系统参数、查看运行状态 | 系统最高业务权限 |
| 实验内容开发者 `developer` | 维护实验模块、Notebook 模板、专项可视化实验配置 | 管理实验模块与模板资源 |

账号策略：

- 不开放公开注册。
- 所有用户账号由管理员创建。
- 用户登录后根据角色看到不同菜单和操作按钮。
- 前端只负责隐藏无权限入口，后端必须在接口层进行真实权限校验。

| 模块 | 前端职责 | 后端职责 |
|------|----------|----------|
| 登录与会话 | 登录表单、Token 保存、请求头注入、登录态过期跳转 | 校验账号密码、签发/刷新/拉黑 Token、返回当前用户 |

| 判题结果 | 轮询或刷新结果、展示状态、stdout/stderr、测试点明细 | Worker 执行代码、计算结果、写入数据库和缓存 |
| 考试系统 | 考试入口、倒计时、提交页面、成绩展示 | 考试 CRUD、时间校验、提交记录、成绩统计 |

| Swagger 对接 | 根据 `/docs` 调试接口，生成请求代码或类型 | 维护 Pydantic Schema、接口标签、响应模型 |

后端必须保证所有敏感操作都有权限校验。前端的菜单隐藏和按钮禁用只是用户体验优化，不能替代后端权限控制。

## 七、统一接口规范

### 7.1 基础约定

| 项目 | 约定 |
|------|------|
| API 前缀 | `/api/v1` |
| 数据格式 | 请求体与响应体均使用 JSON |
| 字段命名 | JSON 字段使用 `snake_case` |
| 时间格式 | ISO 8601 字符串，例如 `2026-06-29T10:30:00+08:00` |
| ID 类型 | 默认使用整数 ID，前端按不透明标识处理 |
| 分页参数 | `page` 从 1 开始，`page_size` 默认 20 |
| 排序参数 | 常用 `sort` 与 `order`，例如 `sort=created_at&order=desc` |
| 认证请求头 | `Authorization: Bearer <access_token>` |
| Swagger 地址 | 后端启动后访问 `/docs` |

### 7.2 统一分页响应

分页列表建议采用以下结构，方便前端组件复用：

```json
{
  "items": [],
  "page": 1,
  "page_size": 20,
  "total": 0
}
```

### 7.3 统一错误响应

后端错误响应建议保持稳定结构：

```json
{
  "detail": {
    "code": "COURSE_NOT_FOUND",
    "message": "课程不存在",
    "fields": {}
  }
}
```

字段说明：

| 字段 | 说明 |
|------|------|
| `code` | 稳定错误码，前端可用于定制提示 |
| `message` | 面向用户或开发调试的中文说明 |
| `fields` | 表单字段错误，可为空对象 |

### 7.4 常用 HTTP 状态码

| 状态码 | 场景 |
|--------|------|
| `200 OK` | 查询、修改、登录成功 |
| `201 Created` | 创建用户、课程、作业、提交成功 |
| `204 No Content` | 删除或登出成功且无需响应体 |
| `400 Bad Request` | 请求参数格式正确但业务规则不满足 |
| `401 Unauthorized` | 未登录、Token 过期或 Token 无效 |
| `403 Forbidden` | 已登录但角色或资源权限不足 |
| `404 Not Found` | 资源不存在 |
| `409 Conflict` | 用户名重复、重复选课等冲突 |
| `422 Unprocessable Entity` | FastAPI 参数校验失败 |
| `500 Internal Server Error` | 服务端未预期错误 |

## 八、认证与权限约定

### 8.1 登录流程

```
前端提交用户名和密码
        │
        ▼
POST /api/v1/auth/login
        │
        ▼
后端校验账号、密码、账号状态
        │
        ▼
返回 access_token、refresh_token、用户信息和角色
        │
        ▼
前端保存 Token，后续请求带 Authorization 请求头
```

登录成功响应建议包含：

```json
{
  "access_token": "string",
  "refresh_token": "string",
  "token_type": "bearer",
  "expires_in": 1800,
  "user": {
    "id": 1,
    "username": "student001",
    "real_name": "张三",
    "role": "student"
  }
}
```

### 8.2 Token 策略

- `access_token` 用于访问业务接口，过期时间较短。
- `refresh_token` 用于刷新登录状态，过期时间较长。
- 登出时后端将当前 Token 加入 Redis 黑名单。
- 刷新 Token 时，旧 Refresh Token 失效，后端返回新的 Access Token 与 Refresh Token。
- 前端遇到 `401` 可尝试刷新 Token；刷新失败则清空登录态并跳转登录页。

### 8.3 角色权限矩阵

| 操作 | student | teacher | admin | developer |
|------|---------|---------|-------|-----------|
| 登录与查看自己信息 | 是 | 是 | 是 | 是 |
| 查看已发布课程 | 是 | 是 | 是 | 是 |
| 创建/编辑课程 | 否 | 是 | 是 | 否 |
| 选课/退课 | 是 | 否 | 是 | 否 |
| 布置作业与考试 | 否 | 是 | 是 | 否 |
| 提交作业与考试 | 是 | 否 | 否 | 否 |
| 查看全班成绩 | 否 | 是 | 是 | 否 |
| 创建用户与分配角色 | 否 | 否 | 是 | 否 |
| 管理实验模块 | 否 | 否 | 是 | 是 |

### 10.1 登录与恢复会话

```
用户打开前端
  -> 前端检查本地是否有 access_token
  -> 若没有 Token，进入登录页
  -> POST /auth/login
  -> 成功后保存 Token 和用户信息
  -> GET /auth/me 校验当前会话
  -> 根据 role 加载对应菜单
```

前端注意事项：

- 不要仅依赖本地缓存的用户角色，刷新页面后应通过 `/auth/me` 恢复可信用户信息。
- 任何接口返回 `401` 时，先尝试 `/auth/refresh`；刷新失败后跳转登录页。
- 任何接口返回 `403` 时，提示无权限，不要重复刷新 Token。

### 10.2 课程浏览与课时学习

```
学生登录
  -> GET /courses 获取可见课程
  -> POST /courses/{course_id}/enroll 选课
  -> GET /courses/{course_id}
  -> GET /courses/{course_id}/chapters
  -> 前端渲染章节和课时内容
```

后端根据用户角色返回不同数据：

- 学生只能看到已发布课程和自己已选课程的完整内容。
- 教师可以看到自己创建或被授权管理的课程。
- 管理员可以查看所有课程。

### 10.3 作业提交与判题结果

```
学生进入作业详情
  -> GET /assignments/{assignment_id}
  -> GET /assignments/{assignment_id}/questions
  -> 前端展示题面、函数签名、公开样例
  -> POST /judge/submissions 提交代码
  -> 后端创建 submission，状态为 queued
  -> Worker 异步判题
  -> 前端轮询 GET /judge/submissions/{id}/result
  -> 展示 accepted / wrong_answer / runtime_error / time_limit_exceeded
```

建议前端轮询策略：

- `queued` 或 `running` 时每 1 到 2 秒查询一次结果。
- 连续查询超过约定次数后提示“判题仍在进行中”，避免无限加载。
- 最终状态出现后停止轮询。

### 10.4 JupyterLab 入口

```
学生点击实验入口
  -> GET /jupyter/entry
  -> 后端校验登录态和课程/实验权限
  -> 返回 iframe_url
  -> 前端在页面 iframe 中加载 /jupyter/* 地址
```

MVP 阶段采用单 JupyterLab 服务：

- 前端只负责 iframe 嵌入与加载错误提示。
- 后端只提供入口 URL、模板列表和权限校验。
- 多用户隔离、独立 Notebook 环境、JupyterHub 接入可作为后续增强。

### 10.5 考试提交与成绩查看

```
学生进入考试
  -> GET /exams/{exam_id}
  -> POST /exams/{exam_id}/start
  -> 前端开始倒计时
  -> 学生提交答案
  -> POST /exams/{exam_id}/submit
  -> 后端记录提交和成绩
  -> 学生查看个人成绩，教师查看班级成绩
```

考试时间校验必须以后端时间为准。前端倒计时只作为展示，不能作为最终有效性判断。

## 十一、判题系统设计

### 11.1 判题题型

MVP 默认只支持 Python 函数测试模式。教师创建题目时提供：

| 字段 | 说明 |
|------|------|
| `title` | 题目标题 |
| `description` | 题面说明，支持 Markdown |
| `function_name` | 学生必须实现的函数名 |
| `signature` | 展示给前端的函数签名，例如 `def add(a: int, b: int) -> int` |
| `starter_code` | 前端代码编辑器默认内容 |
| `public_cases` | 公开样例，用于展示和简单调试 |
| `hidden_tests` | 隐藏 pytest 测试，由后端保存，不返回给学生 |
| `time_limit_ms` | 单次判题超时时间 |
| `memory_limit_mb` | 容器内存限制 |

### 11.2 判题流程

```
POST /judge/submissions
  -> 后端校验学生是否有提交权限
  -> 写入 submissions 表，状态 queued
  -> Redis LPUSH judge:queue {submission_id}
  -> API 立即返回 submission_id 和 queued 状态

Worker BRPOP judge:queue
  -> 查询 submission、question、hidden_tests
  -> 创建临时工作目录
  -> 写入学生代码和 pytest 测试文件
  -> 启动 Docker 容器执行 pytest
  -> 收集 stdout、stderr、退出码、耗时
  -> 更新 submissions 表和 Redis 结果缓存
```

### 11.3 沙箱限制

| 限制 | 默认值 | 说明 |
|------|--------|------|
| 网络 | 禁止 | 防止访问外部资源 |
| CPU | 1 核 | 防止占满主机资源 |
| 内存 | 256 MB | 防止内存滥用 |
| 超时 | 10 秒 | 超时后强制终止 |
| 文件系统 | 只读根文件系统 + 临时工作目录 | 降低破坏风险 |
| 容器生命周期 | 每次提交临时创建，用完删除 | 避免提交之间互相影响 |

### 11.4 判题状态

| 状态 | 说明 | 是否终态 |
|------|------|----------|
| `queued` | 已提交，等待 Worker 消费 | 否 |
| `running` | Worker 正在执行 | 否 |
| `accepted` | 所有测试通过 | 是 |
| `wrong_answer` | 测试未通过 | 是 |
| `runtime_error` | 代码运行异常 | 是 |
| `time_limit_exceeded` | 执行超时 | 是 |
| `system_error` | Worker、Docker 或平台异常 | 是 |

| 类型 | 枚举值 |

| 用户角色 | `student`、`teacher`、`admin`、`developer` |

前端应避免硬编码中文状态文案。建议前端维护状态到展示文案的映射，接口只返回稳定英文枚举值。

## 十四、JupyterLab 集成约定

MVP 阶段采用单 JupyterLab 服务，主要目标是让学生可以在平台内打开 Notebook 完成交互式实验。

### 14.1 前端处理

- 在实验页面中通过 iframe 加载后端返回的 `iframe_url`。
- iframe 外层需要提供加载中、加载失败、刷新按钮。
- 前端不直接拼接 Jupyter 地址，统一通过 `/api/v1/jupyter/entry` 获取。
- 如果后端返回 `403`，说明当前用户没有进入该实验的权限。

### 14.2 后端处理

- 校验用户登录态和实验访问权限。
- 返回当前环境下可用的 Jupyter iframe URL。
- 返回 Notebook 模板列表，供前端展示“从模板开始实验”。
- 记录学生打开实验、复制模板、完成实验等行为。

### 14.3 后续增强

后续如果需要更强隔离，可以从单 JupyterLab 升级为 JupyterHub 或按用户创建容器。该增强不影响前端主流程，只需要保持 `/jupyter/entry` 返回结构稳定。

## 十六、前后端联调清单

后端完成阶段性开发后，应至少满足以下联调条件：

- 后端服务启动成功，`/docs` 可访问。

- 管理员账号可以通过 CLI 或初始化脚本创建。
- 前端可以完成登录、刷新登录态、登出。
- 前端可以根据 `/auth/me` 的角色信息渲染菜单。
- 学生可以获取课程列表、课程详情、章节课时。
- 教师或管理员可以创建作业和判题题目。
- 学生可以提交 Python 函数代码并查询判题结果。
- Jupyter 页面可以通过后端返回的 URL 正常嵌入。
- 错误响应格式稳定，前端可以统一弹出提示。
